# 연구노트 (책임자 관점)

- **날짜**: 2025-02-05
- **작성자**: 강혜림
- **역할**: 책임연구원
- **진행 단계**: 3단계 서비스 통합 점검·고도화
- **아젠다**: 자동화 리팩터링 Phase 2 착수, 코드 스캐폴딩·캐드, QA 전략 정비

---

## 1. 연구 배경·목적 (책임 관점)

- IRIS 업로드·교육·모니터링 자동화를 1차로 구축한 이후, **Phase 2 리팩터링**을 통해 구조를 정리하고 장기 유지보수 비용을 줄이는 것이 목적.
- 공통 모듈화·테스트 용이성·배포 안정성을 동시에 끌어올릴 수 있도록 설계 단계부터 책임자 관점에서 방향을 잡고자 함.

---

## 2. 오늘 주요 진행 내용

### 2.1 Phase 2 구조·역할 정리
- 자동화 스크립트들의 공통 부분을 분리하여, `scripts/automation/refactor_phase2/` 아래로 모으는 구조 제안.
- 운영·개발·QA 각자의 역할을 **RACI 기반**으로 정리해, 누가 어떤 스크립트·배포·모니터링에 책임을 갖는지 문서화.

### 2.2 코드 스캐폴딩 및 캐드 작성
- `scripts/automation/refactor_phase2/` 디렉터리 생성 및 기본 스캐폴딩 코드 2개 작성.
- 기존 스크립트에서 가장 공통적으로 사용하는 로깅·리트라이·구성 로딩 부분을 Phase 2 공통 모듈에 흡수하는 방향을 잡음.

### 2.3 QA 전략 구체화
- 단위 테스트/통합 테스트를 구분하여, 최소한의 회귀 보장을 줄 수 있는 테스트 세트를 정의.
- GitHub Actions 등 CI 파이프라인에 Phase 2 관련 테스트를 어떻게 통합할지 매트릭스 수준에서 설계.

### 2.4 커뮤니케이션
- Slack/Teams를 통해 Phase 2 목표·RACI 요약·디렉터리 구조를 공유하여, 이후 리뷰·기여가 수월하도록 안내.

## 4. 책임자 관점 의사결정·리스크

- **의사결정**
  - Phase 2에서는 기능 추가보다는 **구조 정리와 품질 확보**에 집중하기로 함.
  - RACI 체계를 명확히 하여, 장애 발생 시 담당자와 백업 담당이 빠르게 식별되도록 함.

- **리스크**
  - 일부 스크립트가 Python 3.10 기능에 의존하고 있어, 실행 환경 업그레이드가 동반되지 않으면 리팩터링 이득이 제한적일 수 있음.
  - QA 인력과 시간 리소스가 제한적이어서, 테스트 범위를 어디까지 가져갈지에 대한 추가 조율이 필요.
