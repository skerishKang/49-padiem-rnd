# 연구노트 (책임자 관점)

- **날짜**: 2025-02-06
- **작성자**: 강혜림
- **역할**: 책임연구원
- **진행 단계**: 3단계 서비스 통합 점검·고도화
- **아젠다**: 교육 파이프라인 튜닝, 의존성 블로커 처리, 자동 테스트·로그 개선

---

## 1. 연구 배경·목적 (책임 관점)

- 교육 파이프라인이 실제 운영 환경에서 안정적으로 동작하도록 **성능 튜닝(캐시 전략)**을 진행하고, 남아 있던 의존성 블로커를 해소해 스프린트 흐름을 매끄럽게 만드는 것이 목적.
- 동시에 자동 테스트 로그 포맷을 개선해, 이후 장애·성능 이슈 분석 시간을 줄이기 위한 기반 작업을 수행.

---

## 2. 오늘 주요 진행 내용

### 2.1 교육 파이프라인 성능 튜닝
- Hot/Cold 캐시 전략을 분리 적용하여, 처리 시간 **7.5분 → 6.8분** 수준으로 단축.
- 리소스 사용량과 처리량 추이를 Grafana 대시보드로 재확인하고, 교육 시간대 트래픽 패턴을 기반으로 캐시 만료 정책을 미세 조정.

### 2.2 블로커 처리
- 교육·IRIS 쪽에서 남아 있던 의존성 블로커 1건을 데이터셋 정합성 점검·보완을 통해 해결.
- 해결 내역을 Jira 이슈에 반영하고, 관련 팀과 재발 방지 방안을 간단히 합의.
- 별도의 외부 의존이 필요한 항목에 대해서는 추가 협의(파트너 콜) 준비.

### 2.3 자동 테스트·로그 개선
- 자동 테스트 실행 로그 포맷을 **JSON Lines** 형태로 통일하는 스크립트를 추가.
- GitHub Actions에서 테스트 결과와 함께 구조화된 로그를 수집할 수 있도록 워크플로를 보완.
- 로그 용량과 보관주기 기준을 다시 정리해, 디스크 사용량과 분석 편의성 사이의 균형점을 검토.

### 2.4 커뮤니케이션
- Slack을 통해 오늘 수행한 교육 튜닝 결과(처리 시간 개선), 블로커 해소 현황, 로그 개선 방향을 공유.
- 향후 교육 일정·트래픽 증가를 고려한 추가 튜닝 여지를 설명.

---

## 3. 산출물·지표

- 교육 파이프라인 처리 시간: **7.5분 → 6.8분** 단축.
- 의존성 블로커 1건 해결, 1건은 후속 협의 대상으로 정리.
- JSON Lines 기반 테스트 로그 생성 스크립트 및 CI 연동 구성.

---

## 4. 책임자 관점 의사결정·리스크

- **의사결정**
  - 교육 파이프라인은 추가 기능보다는 **성능·안정성**에 우선순위를 두고, 캐시·로그 튜닝을 먼저 마무리하기로 결정.
  - 블로커는 Jira 상에서 명시적으로 상태를 관리하고, 스프린트 플래닝 시 "해결 기한"을 함께 정의해 두는 방식으로 운영.

- **리스크**
  - Hot 캐시 초기 로딩 문제는 Warm-up 스크립트가 정착되기 전까지 일시적인 장애로 이어질 수 있음.
  - 외부 파트너 의존성이 남아 있어, 일정(2/7 파트너 콜 등)에 따라 일부 과제 착수 시점이 미뤄질 수 있음.
